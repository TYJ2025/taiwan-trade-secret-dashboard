# Taiwan Trade Secrets Case Tracker — Agent 工作守則

本檔為本 repo 內所有 Claude（或其他 LLM agent）執行任務時必須遵循的規範。
作者 YJ 為執業 25 年律師，專案最終讀者為法律專業人士，**資料正確性、可稽核性、誠實揭露限制**為三大首要原則。

---

## 1. Log-first 原則（最高優先）

**任何會改動檔案、執行 bash／npm／python、或對外抓取資料的行為，都必須先寫入 `SESSION_LOG_<YYYY-MM-DD>.md` 才開始執行。**

- 一個工作日對應一份 log；同一天多個 session 接續寫入同一檔，以 `## Session HH:MM` 區隔。
- Log 必須在「動手前」記下「**意圖（為什麼）+ 預期結果（成功長什麼樣子）**」，動手後再補「**實際結果 + 異常**」。
- 違反此原則的成果（即使技術上正確）視為未驗證，需重做。

### 每筆 log 條目最低欄位

```
### [HH:MM] <動作標題>
- 意圖：<為什麼要做、要解決什麼問題>
- 指令：<完整 bash 指令或檔案修改摘要>
- 預期結果：<成功長什麼樣子；數量、檔名、HTTP code…>
- 實際結果：<指令輸出的關鍵摘要；不要貼整段 stdout>
- 異常／差異：<與預期不符之處；無則寫「無」>
- 後續行動：<是否需修正；連結至下一個條目>
```

範本見 `SESSION_LOG_TEMPLATE.md`（repo 根目錄）；新 session 直接 `cp SESSION_LOG_TEMPLATE.md SESSION_LOG_$(date +%Y-%m-%d).md` 即可。

## 2. 資料變更必須抽樣驗證

新增或修改 `data/*.json`、`trade_secret_judgments_*.json`、或任何給前端讀的資料，**必須**：

1. 在 log 中列出**至少 3 筆抽樣案件**的 before / after 比對（如金額、計算方式、outcome）。
2. 抽樣須涵蓋已知的標竿案件（例：大立光 vs 先進光，智財法院 102 民營訴 6 / 107 民營上 1）。
3. 若新增金額抽取邏輯，必須驗證：(a) 大立光案 = 1,522,470,639 元；(b) 隨機挑一筆「原告敗訴」案件，其判准金額為 0。
4. 統計類輸出（總額、件數）若與直覺差距 > 1 個數量級，**先停下來在 log 中質疑**，不要直接接受。本次最初 784 億 vs 修正後 33 億即為實例（見 `SESSION_LOG_2026-04-18.md` §2 Run 1 與 Run 2）。

---

## 3. 誠實揭露已知限制

**最終交付物必須在程式或頁面內明示限制**，例如：

- DamagesAnalysis.jsx 末段已有「資料抽取說明」段落 — 任何規則修改後，同步更新該段。
- 若新增啟發式或正則抽取，需在 log 寫明 false positive / negative 的可能來源與比例。
- 不可在使用者介面用「精確」「百分百」「完整」等絕對性詞彙描述自動化抽取結果；用「以正則比對」「啟發式偵測」「以全文比對」等中性語。

---

## 4. 不破壞既有部署

- `index.legacy.html` 為舊版單檔 Chart.js 儀表板的封存，**永不刪除**，除非 YJ 明確同意。
- `data/cases.json` 與 `data/stats.json` 屬獨立的「審理中／訴訟前」資料集（52 筆），不可與 492 筆判決資料混淆或互相覆蓋。
- 修改 `vite.config.js` 的 `base` 路徑前，必須先確認 GitHub Pages 站點 URL；本專案固定為 `/taiwan-trade-secret-dashboard/`。
- 修改 `.github/workflows/*.yml` 必須在 log 中說明變更原因，並在下次 push 前模擬一次 dry-run。

---

## 5. Python 抽取腳本維護準則

- 所有抽取邏輯（`extract_fields.py`, `extract_damages.py`）都須保留 sanity-check assertions（如 `assert cn_to_int_mixed('壹拾伍億…') == 1_522_470_639`）。
- 新增 regex 必須附 inline comment 說明對應之法條或實務用語（例：`# 民§216 II 所失利益`）。
- 抽取完成後，需印出至少 5 行的「TOP 10 結果」摘要，方便事後肉眼複核（已實作於 `extract_damages.py` main()）。

---

## 6. 法律術語規範（YJ 校稿用）

- 一律使用「**判准金額**」（court-awarded）vs「**請求金額**」（plaintiff-claimed），不要寫「賠償金額」之類含糊用語。
- 「損害賠償案件」定義固定為：民事案件且案由含「損害賠償／賠償／排除侵害」之一；刑事附帶民事另列。
- 條文引用格式統一：`營業秘密法§13 I (1)`、`民§184`、`民訴§222 II`。簡寫表已在 `extract_damages.py` 的 `STATUTE_CHECK` 集中維護，新增條文須同步更新該表。

---

## 7. 對話／互動規範

- YJ 偏好繁體中文回覆（除非明確要求英文）。
- 思考可以用英文，但最終答覆與所有檔案內容需為繁體中文。
- 給 YJ 的回覆中，技術細節務求精簡；金額／條文／案號務求精確。

---

## 8. 違反守則之處置

若 agent 在 session 中偏離本守則（例：未先寫 log 即執行 bash），應**主動停下**並：

1. 在 log 中補上「**遲報條目**」說明已執行之行為；
2. 說明違反原因（例：「忘了寫 log」「以為是唯讀指令」）；
3. 評估是否需 rollback；若資料已寫入但未驗證，必須先驗證再繼續。
